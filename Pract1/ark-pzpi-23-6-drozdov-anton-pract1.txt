2 ЗАВДАННЯ
Метою цієї роботи є вивчення рекомендацій щодо оформлення програмного коду мовою Асемблера відповідно до Assembly Language Coding Guidelines. А також дослідження існуючих інструментів для форматування коду написаного мовою Асемблера.

3 ОПИС ВИКОНАНОЇ РОБОТИ
        3.1 Вибір мови програмування та джерел
        Для виконання практичної роботи було обрано мову Асемблера, у практичній роботі розглядаються правила з використанням мови асемблеру AT&T синтаксису під x86, але правила загальні та поширюються на будь-який діалект та платформу. Для дослідження було обрано Assembly Language Coding Guidelines від ACRN Project у якості правил оформлення коду.
        3.2 Виконання роботи
        Assembly Language Coding Guidelines розбиті на 5 секцій: Загальні правила (General), Функції (Functions), Стиль кодування (Coding Style), Найменування (Naming Style) та ABI Відповідність (ABI Conformance).
        Було розглянуто усі правила та вибрано по декілька правил із кожної секції, окрім секції ABI Відповідність, оскільки ця секція має лише одне правило, яке говорить, що код, власне, повинен відповідати бінарному інтерфейсу платформи.
        Розглянемо наступні правила:
ASM-GN-01: Одна адреса не повинна бути вказана двома мітками
1. /* Добре - одна мітка на одну адресу */
2. asm_showcase_1:
3.    movl    $0x1, %eax
4. 5. /* Погано - дві мітки на одну і ту ж саму адресу */
6. asm_showcase_1:
7. asm_showcase_2:
8.        movl    $0x1, %eax




ASM-GN-02: Не використовувати зарезервовані імена
1. /* Добре - зарезервовані слова не використовуються */
2. asm_showcase_1:
3.        movl    $0x1, %eax
4. 5. asm_showcase_2:
6.        movl    $0x2, %eax
7. 8. asm_showcase_3:
9.        movl    $0x3, %eax
10. 11. /* Погано - використовуються зарезервовані слова text, mov та eax */
12. text:
13.        movl    $0x1, %eax
14. 15. mov:
16.        movl    $0x2, %eax
17. 18. eax:
19.        movl    $0x3, %eax


ASM-GN-05: Використовувати дужки для вказування пріоритету операторів
1. /* Добре - завдяки дужкам очевидно, який порядок операцій */
2. .section .data
3. showcase_data:
4.        /* 0x1234 means <something> */
5.        .long   0x1234 * (0x1234 >> 2)
6. 7. /* Погано - не можна відразу сказати порядок операцій */
8. .section .data
9. showcase_data:
10.        /* 0x1234 means <something> */
11.        .long   0x1234 * 0x1234 >> 2


ASM-FN-01: Функції повинні мати оператор return
1. /* Добре - функція має оператор return */
2. asm_func_showcase:
3.        movl    $0x2, %eax
4.        ret
5. 6. asm_showcase:
7.        movl    $0x1, %eax
8.        call    asm_func_showcase
9. 10. /* Погано - немає явного оператору return */
11. asm_func_showcase:
12.        movl    $0x2, %eax
13. 14. asm_showcase:
15.        movl    $0x1, %eax
16.        call    asm_func_showcase


ASM-FN-02: У функції повинна бути лише одна точка входу
1. /* Добре - мітка tmp використовується лише всередині функції */
2. asm_func_showcase:
3.        test    $0x400, %eax
4.        jne     tmp
5.        movl    $0x1, %eax
6. tmp:
7.        movl    $0x2, %eax
8.        ret
9. 10. asm_showcase_good:
11.        movl    $0x1, %eax
12.        /* Добре - сама функція викликається за допомогою call */
13.        call    asm_func_showcase
14. 15. asm_showcase_bad:
16.        movl    $0x1, %eax
17.        call    asm_func_showcase
18.        /* Погано - стрибок усередину функції через мітку */
19.        jmp     tmp
ASM-CS-01: Одна інструкція - один рядок
1. /* Добре - один рядок на інструкцію */
2. movl    $0x2, %eax
3. 4. /* Погано - інструкція розділена на два рядки */
5. movl    $0x2, \
6. %eax


ASM-NC-02: Імена називаються у lower_snake_case
1. /* Добре - використовується snake case у нижньому регістрі */
2. asm_showcase:
3.        movl    $0x1, %eax
4. 5. /* Погано - використовується верхній регістр */
6. ASM_SHOWCASE:
7.        movl    $0x1, %eax


ASM-NC-04: Імена не повинні бути більше 31 символа
1. /* Добре - коротка назва */
2. asm_showcase:
3.        movl    $0x1, %eax
4. 5. /* Погано - назва більше 31 символа */
6. asm_showcase_asm_showcase_asm_showcase:
7.        movl    $0x1, %eax


        Також були досліджені існуючі інструменти для форматування коду на асемблері. Таких інструментів не дуже багато. Основний сучасний інструмент - це ASMfmt. Також існують розширення у Visual Studio Code, які можуть форматувати асемблерний код. Clang-format може форматувати асемблерні вставки.
        Після дослідження було сформовано презентацію та записано відеозапис доповіді.

4 ВИСНОВКИ
        Під час виконання практичної роботи було досліджено Coding Guidelines для мови Асемблеру від ACRN Project. Було досліджено інструменти для форматування коду на мові Асемблеру. Також було ознайомлено з класифікацією мов асемблеру за діалектом та цільовою архітектурою, зі сферами застосування асемблеру у сучасному світі та у минулому.
